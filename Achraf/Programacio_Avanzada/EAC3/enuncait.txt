Exercici 1:  Serveis: HTTP + estats de càrrega / èxit / error (2,5 punts)
Enunciat: Implementeu el servei ElementService al fitxer src/app/serveis/element.service.ts. Aquest servei és el nucli de comunicació entre l'aplicació i l'API mock, i ha de gestionar els tres estats de manera reactiva amb signals.
El servei ha d'exposar les signals de lectura següents:
    • elements: Signal<ElementCataleg[]> — llista d'elements carregats des de l'API.
    • carregant: Signal<boolean> — true mentre hi ha una petició HTTP en curs.
    • error: Signal<string | null> — missatge d'error descriptiu, o null si tot va bé.

El servei ha d'implementar els mètodes:
    • obtenirPopulars(): void — carrega els elements amb popular=true des de l'API.
    • cercar(terme: string): void — filtra elements per nom consultant l'API.
Requisit important: cada mètode ha de gestionar els tres estats: activar carregant=true en iniciar la petició, actualitzar elements en cas d’èxit, i establir un missatge d'error comprensible en cas de fallada.
Heu de configurar provideHttpClient() a app.config.ts i definir la URL base de l'API en un fitxer d'entorn (src/environments/environment.ts).
El component CatalegPage ha de mostrar els tres estats diferenciats visualment:
    • Carregant: spinner animat mentre l'API respon.
    • Error: missatge clar i botó «Reintentar».
    • Èxit: graella responsiva amb les targetes dels elements.

Captura requerida 1: Estat carregant — spinner visible mentre l'API respon (retard de 600 ms del json-server).
Captura requerida 2: Estat error — missatge d'error i botó «Reintentar» visibles (atureu el json-server per simular-ho).
Captura requerida 3: Estat èxit — graella d'elements carregats correctament des de l'API.

Exercici 3: Formularis: Reactius + validacions sync/async (2 punts)
Enunciat: Implementeu el component FormulariCerca al fitxer src/app/components/formulari-cerca/. Aquest component ha de gestionar la cerca d'elements amb un formulari reactiu i validacions avançades.
El formulari ha d'incloure el camp termeCerca amb:
    • Validador síncron: longitud mínima de 2 caràcters i màxima de 50.
    • Validador asíncron codiDisponibleValidator: simula una consulta a l'API per comprovar si hi ha resultats (retard de 500 ms). Ha de retornar { sensResultats: true } si no hi ha elements.

Comportament del component:
    • Cerca automàtica amb debounce de 400 ms mentre l'usuari escriu.
    • Botó «Netejar» que apareix únicament quan el camp conté text.
    • Indicador visual («Validant...» o spinner) durant la validació asíncrona.
    • Missatges d'error visibles per a cadascuna de les validacions.
    • Els errors no s'han de mostrar fins que l'usuari hagi tocat el camp (ng-touched).

Documentació obligatòria a docs/formularis.md: descriviu els validadors implementats, síncrons i asíncrons, i el comportament del debounce.
